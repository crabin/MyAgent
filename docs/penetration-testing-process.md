# 标准渗透测试流程

本文档整理标准授权渗透测试流程。默认前提是：已获得书面授权、明确测试范围、测试窗口、限制条件和应急联系人。

## 1. 资产发现

目标：确认测试范围内有哪些资产、服务、系统、域名、IP、云资源和第三方暴露面。

| 工具/方法 | 作用 |
| --- | --- |
| 客户提供资产清单 | 明确授权范围，避免越界 |
| Whois / RDAP | 查询域名注册信息、组织信息、关联邮箱 |
| DNS 枚举 | 发现子域名、MX、NS、TXT、CNAME 等记录 |
| Subfinder | 被动子域名收集 |
| Amass | 子域名发现、组织资产关联分析 |
| Assetfinder | 快速收集关联域名和子域名 |
| SecurityTrails | 查询历史 DNS、子域、IP 关联 |
| Censys | 搜索互联网暴露服务、证书、端口 |
| Shodan | 发现公网暴露主机、端口、Banner |
| FOFA / ZoomEye | 互联网空间测绘，适合国内外资产发现 |
| crt.sh | 通过证书透明日志发现子域 |
| GitHub / GitLab 搜索 | 查找泄露的域名、密钥、配置、内部地址 |
| 云资产盘点 | 检查 AWS、Azure、GCP、阿里云、腾讯云暴露资源 |
| IP 反查 | 从域名解析到 IP，再扩展同网段资产 |
| ASN 查询 | 根据组织 ASN 发现 IP 段 |

## 2. 攻击面分析

目标：理解每个资产暴露了什么服务、入口、技术栈、认证方式和潜在风险点。

| 工具/方法 | 作用 |
| --- | --- |
| Nmap | 端口扫描、服务识别、操作系统探测 |
| Masscan | 大范围高速端口探测 |
| httpx | 批量探测 HTTP/HTTPS 服务状态、标题、技术栈 |
| Naabu | 快速端口扫描，常用于批量资产 |
| WhatWeb | Web 指纹识别 |
| Wappalyzer | 识别前端框架、CMS、中间件、分析组件 |
| BuiltWith | 分析网站技术栈 |
| EyeWitness | 批量截图，辅助人工判断 Web 入口 |
| Aquatone | 子域截图和 Web 服务可视化 |
| Dirsearch | Web 目录和路径枚举 |
| FFUF | 高性能目录、参数、虚拟主机枚举 |
| Gobuster | 目录、DNS、VHost 爆破 |
| Katana | Web 爬虫，收集 URL、接口、参数 |
| Hakrawler | 快速爬取 Web 链接 |
| JS 分析 | 从前端 JS 中发现 API、密钥、隐藏路径 |
| robots.txt / sitemap.xml | 发现隐藏路径和索引页面 |
| Swagger / OpenAPI | 发现 API 文档和接口定义 |
| GraphQL introspection | 检查 GraphQL Schema 是否暴露 |

| Nikto | Web 服务常见漏洞和错误配置检查 |
| Burp Suite Scanner | Web 应用自动化漏洞扫描 |
| OWASP ZAP | 开源 Web 漏洞扫描代理 |
| Acunetix | 商业 Web 漏洞扫描器 |
| Xray | Web 漏洞扫描工具 |
| SQLMap | SQL 注入检测和验证 |
| WPScan | WordPress 漏洞、插件、用户枚举 |
| CMSMap | CMS 漏洞检测 |
| Trivy | 容器镜像、依赖、IaC 漏洞扫描 |
| Grype | 软件包和容器漏洞扫描 |
| Syft | 生成 SBOM，辅助依赖风险分析 |
| kube-bench | Kubernetes CIS 基线检查 |

| ScoutSuite | 云环境安全配置审计 |

| Snyk | 依赖、容器、IaC 安全扫描 |
| Semgrep | 静态代码安全规则扫描 |
| CodeQL | 代码级漏洞分析 |

## 4. 人工验证

目标：确认扫描结果是否真实可利用，排除误报，评估实际影响。

| 方法 | 作用 |
| --- | --- |
| 请求重放 | 使用 Burp Repeater 验证漏洞触发条件 |
| 参数篡改 | 验证越权、IDOR、逻辑漏洞 |
| 权限边界测试 | 使用不同角色账号验证访问控制 |
| 输入点测试 | 检查 XSS、SQL 注入、命令注入、模板注入 |
| 业务流程测试 | 检查支付、审批、注册、找回密码等逻辑缺陷 |
| Session 测试 | 检查 Cookie、安全属性、会话固定、失效机制 |
| 认证测试 | 检查 MFA、弱密码策略、验证码、登录限制 |
| 文件上传测试 | 检查类型校验、路径穿越、恶意文件处理 |
| API 测试 | 检查鉴权、限流、参数校验、批量操作 |
| 日志和错误信息分析 | 发现堆栈信息、内部路径、敏感配置 |
| 影响复现 | 证明漏洞导致的数据读取、权限变化或业务影响 |

常用工具：

| 工具 | 作用 |
| --- | --- |
| Burp Suite | Web 手工测试核心代理工具 |
| OWASP ZAP | Web 代理、拦截、重放 |
| Postman | API 请求构造和验证 |
| Insomnia | API 测试客户端 |
| curl / HTTPie | 命令行请求验证 |
| Browser DevTools | 前端请求、存储、CORS、JS 调试 |
| JWT.io | 分析 JWT 结构和声明 |
| CyberChef | 编码、解码、加解密辅助 |
| jq | JSON 数据处理 |

## 5. 漏洞利用

目标：在授权范围内证明漏洞真实影响，避免破坏性操作。

| 类型 | 方法/工具 | 作用 |
| --- | --- | --- |
| Web 漏洞 | Burp Suite | 构造利用请求、验证影响 |
| SQL 注入 | SQLMap / 手工验证 | 验证注入点和数据访问风险 |
| XSS | 浏览器、Burp | 验证脚本执行、Cookie 风险、钓鱼风险 |
| SSRF | Burp Collaborator / interactsh | 验证服务端外连和内网访问能力 |
| XXE | 自建回连服务 / Collaborator | 验证 XML 外部实体解析 |
| RCE | 安全 PoC | 验证命令执行风险，避免破坏性命令 |
| 文件上传 | WebShell 模拟文件 / 受控测试文件 | 验证上传绕过和执行风险 |
| 反序列化 | ysoserial 等测试工具 | 验证危险对象反序列化风险 |
| 模板注入 | 手工 payload | 验证服务端模板执行风险 |
| 默认口令 | Hydra / Medusa / Ncrack | 验证弱口令和爆破风险 |
| 已知 CVE | Metasploit / Nuclei / Exploit-DB PoC | 验证公开漏洞影响 |
| API 越权 | 多账号对比 | 验证横向、纵向越权 |
| 云配置缺陷 | 云 CLI / ScoutSuite / Prowler | 验证存储桶、IAM、密钥风险 |

实际项目中，漏洞利用应以“证明影响”为目标，不应扩大破坏面。例如只读取测试数据、执行无害命令、创建临时标识文件、使用客户提供的测试账号。

## 6. 权限提升

目标：在已获得低权限访问的情况下，评估是否能提升为更高权限。

| 场景 | 工具/方法 | 作用 |
| --- | --- | --- |
| Linux 本地提权 | LinPEAS | 自动检查提权线索 |
| Linux 本地提权 | Linux Exploit Suggester | 根据内核和配置匹配潜在漏洞 |
| Windows 本地提权 | WinPEAS | 检查服务、权限、凭据、配置问题 |
| Windows 本地提权 | Seatbelt | 收集 Windows 安全配置和敏感信息 |
| Windows 本地提权 | PowerUp | 检查服务权限、计划任务等提权点 |
| AD 环境 | BloodHound | 分析域内权限关系和攻击路径 |
| AD 环境 | SharpHound | 采集 AD 关系数据 |
| AD 环境 | PowerView | 域信息枚举和权限分析 |
| 凭据收集 | LaZagne | 查找本地保存的凭据 |
| 错误配置 | sudo -l / 服务权限 / SUID | 检查可滥用权限配置 |
| 云环境 | IAM 权限分析 | 检查是否可扩大云账号权限 |
| 容器环境 | Capabilities / 挂载检查 | 判断容器逃逸或宿主机访问风险 |
| Kubernetes | RBAC 检查 | 判断 ServiceAccount 是否权限过大 |

常见提权思路：

| 方法 | 作用 |
| --- | --- |
| 内核漏洞 | 利用系统未修补漏洞提升权限 |
| 弱文件权限 | 修改服务、脚本、配置获取高权限执行 |
| SUID/SGID 滥用 | 利用错误设置的特权二进制 |
| sudo 配置错误 | 利用免密或危险命令获得高权限 |
| 计划任务滥用 | 修改高权限定时任务执行内容 |
| 服务权限错误 | 替换服务二进制或参数 |
| 明文凭据 | 从配置、历史命令、环境变量中发现密码 |
| Token / Session 滥用 | 利用有效会话或令牌横向移动 |
| 过度 IAM 权限 | 在云环境中通过策略组合提升权限 |

## 7. 攻击链组合

目标：把多个低中风险问题组合成真实业务风险路径。

| 攻击链 | 说明 |
| --- | --- |
| 子域接管 -> 钓鱼 -> 凭据泄露 | 低危资产问题组合成账号风险 |
| 信息泄露 -> API 越权 -> 数据导出 | 通过前端 JS 或接口文档发现隐藏 API |
| SSRF -> 云元数据访问 -> 云凭据泄露 | 云环境常见高危链路 |
| 文件上传 -> WebShell -> 内网探测 | 上传缺陷导致服务器控制风险 |
| SQL 注入 -> 数据库权限 -> 文件写入 | 数据库权限过大时可能扩大影响 |
| 弱口令 -> VPN 登录 -> 内网横向移动 | 外部入口弱口令引发内网风险 |
| Git 泄露 -> 配置密钥 -> 后台登录 | 源码或配置泄露导致业务系统失陷 |
| XSS -> 管理员会话 -> 后台操作 | 存储型 XSS 对高权限用户影响更大 |
| Jenkins 弱口令 -> 构建脚本执行 -> 主机权限 | CI/CD 系统常见攻击链 |
| Kubernetes Dashboard 暴露 -> Pod 创建 -> 集群权限 | 容器编排环境攻击链 |

常用方法：

| 方法 | 作用 |
| --- | --- |
| Kill Chain 建模 | 按侦察、投递、利用、控制、横向移动分析 |
| ATT&CK 映射 | 将发现的问题映射到攻击技术 |
| 权限路径图 | 展示从初始入口到高价值资产的路径 |
| 数据流分析 | 找出漏洞对敏感数据的影响 |
| 业务影响建模 | 把技术漏洞转化为业务损失描述 |

## 8. 风险评估

目标：判断漏洞严重程度、影响范围、利用难度和修复优先级。

| 方法/标准 | 作用 |
| --- | --- |
| CVSS | 标准化漏洞评分 |
| OWASP Risk Rating | Web 应用风险评级 |
| DREAD | 从破坏性、复现性、可利用性等维度评分 |
| STRIDE | 威胁建模分类：伪造、篡改、抵赖、信息泄露、拒绝服务、权限提升 |
| MITRE ATT&CK | 关联真实攻击技术和路径 |
| 业务影响评估 | 结合数据敏感性、系统重要性、合规要求评估 |
| 可利用性评估 | 判断是否需要认证、是否公网可达、是否有公开 PoC |
| 资产重要性评估 | 核心系统、生产环境、客户数据优先级更高 |
| 横向移动能力 | 判断是否能从单点问题扩大到内网或云环境 |
| 检测难度 | 判断攻击是否容易被日志和监控发现 |

风险评级常见维度：

| 维度 | 说明 |
| --- | --- |
| 影响 | 是否导致数据泄露、系统控制、业务中断 |
| 范围 | 单个用户、单个系统、全局系统、内网 |
| 利用难度 | 是否需要账号、交互、特殊条件 |
| 可复现性 | 是否稳定触发 |
| 暴露面 | 是否公网可访问 |
| 权限要求 | 是否匿名可利用 |
| 修复复杂度 | 是否需要架构调整 |
| 合规影响 | 是否涉及隐私、金融、医疗、等保、PCI 等 |

## 9. 修复建议

目标：给出可执行、可验证、按优先级排序的修复方案。

| 漏洞类型 | 修复建议 |
| --- | --- |
| SQL 注入 | 使用参数化查询、ORM 安全 API、输入校验、最小化数据库权限 |
| XSS | 输出编码、CSP、输入过滤、HttpOnly Cookie |
| CSRF | CSRF Token、SameSite Cookie、关键操作二次确认 |
| SSRF | URL 白名单、禁止访问内网地址、禁用重定向、网络隔离 |
| RCE | 禁止拼接命令、使用安全 API、最小权限运行服务 |
| 文件上传 | 限制后缀、校验 MIME、重命名文件、隔离存储、禁止执行权限 |
| 越权 | 服务端强制鉴权、对象级权限校验、统一权限中间件 |
| 弱口令 | 强密码策略、MFA、登录限速、账号锁定 |
| 敏感信息泄露 | 移除密钥、轮换凭据、关闭调试、清理历史提交 |
| 默认配置 | 禁用默认账号、修改默认口令、关闭不必要服务 |
| 过度权限 | 最小权限原则、细化 IAM / RBAC 权限 |
| 组件漏洞 | 升级依赖、移除废弃组件、建立补丁流程 |
| TLS 问题 | 禁用弱协议和弱加密套件，启用 HSTS |
| CORS 错误 | 限制 Origin，禁止任意来源携带凭据 |
| 云存储桶公开 | 关闭公开读写、启用访问日志、最小化 Bucket Policy |
| Kubernetes 风险 | 限制 RBAC、禁用特权容器、启用 NetworkPolicy |
| 日志不足 | 增加认证、权限、敏感操作日志和告警 |

修复建议应包含：

| 内容 | 说明 |
| --- | --- |
| 漏洞描述 | 问题是什么 |
| 影响说明 | 可能造成什么损失 |
| 复现证据 | 截图、请求、响应、日志 |
| 修复步骤 | 开发或运维可执行 |
| 优先级 | 高危优先修复 |
| 临时缓解 | 在完整修复前降低风险 |
| 验证方式 | 修完后如何确认有效 |

## 10. 复测

目标：确认漏洞已修复，且没有引入新问题。

| 方法/工具 | 作用 |
| --- | --- |
| 原始 PoC 重放 | 验证漏洞是否仍可触发 |
| Burp Repeater | 重放原请求，对比修复前后响应 |
| 自动扫描复测 | 使用 Nessus、Nuclei、ZAP 等确认漏洞消失 |
| 权限回归测试 | 验证不同角色不能越权 |
| 单元/集成测试 | 固化修复逻辑，防止回归 |
| 代码审查 | 检查修复是否彻底 |
| 配置审计 | 确认安全配置已生效 |
| 日志验证 | 确认攻击尝试能被记录和告警 |
| 旁路测试 | 尝试换参数、换接口、换角色绕过修复 |
| 回归扫描 | 检查修复是否引入新暴露面 |

复测输出一般包括：

| 内容 | 说明 |
| --- | --- |
| 漏洞编号 | 对应报告中的问题 |
| 修复状态 | 已修复、部分修复、未修复、无法验证 |
| 复测证据 | 请求响应、截图、日志 |
| 剩余风险 | 是否仍存在低风险残留 |
| 后续建议 | 是否需要架构调整或长期治理 |

## 简化流程

```text
授权与范围确认
  -> 资产发现
  -> 攻击面分析
  -> 漏洞扫描
  -> 人工验证
  -> 有限漏洞利用
  -> 权限提升评估
  -> 攻击链组合
  -> 风险评级
  -> 报告与修复建议
  -> 复测
  -> 关闭问题
```

核心原则：自动化工具负责覆盖面，人工验证负责准确性，攻击链分析负责业务影响，复测负责闭环。
